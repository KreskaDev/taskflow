# ADR-0008 — Notifications

**Status:** Accepted (2026-06-14)
**Builds on:** ADR-0001 (Wolverine + domain events + transactional outbox), ADR-0003 (domain
model; events as the change signal), ADR-0007 (Real-time: the SignalR transport, which
explicitly anticipates that "Notifications (US-16) reuse the same transport but are specified
in their own ADR"), and constitution v3.0.0 (Architecture & Stack: "an in-app notification
center plus live SignalR toasts, generated by domain-event handlers").
**Scope:** how in-app notifications are generated, stored, delivered, and managed. Realizes
US-16 and FR-079..FR-084; introduces the **Notification** entity (ENT-09).

## Context

The collaborative product (constitution v3.0.0) needs to close the loop on collaboration:
a member must learn when something needs their attention — when they are assigned a task,
@mentioned in a comment, or when a task they are an assignee of changes. The triggering
acts already exist as domain events in other slices (assignment in slice 008, comments and
mentions in slice 009, and task-mutation events published throughout core Task Management —
e.g. slices 002/005/010). Notifications are a **consumer** of those events, not a new source
of truth.

Two constraints shape the decision. First, **in-app only** — email and push/device
notifications and reminders are explicitly out of scope (ASM-06, OOS-13, OOS-14, FR-084);
there is no external notification transport to integrate. Second, the real-time channel
already exists: the SignalR transport decided in ADR-0007 (US-15) pushes live updates to
members viewing shared projects, so a new notification can ride the same connection as a live
toast without new infrastructure.

## Decisions

1. **Notification entity (ENT-09).** A `Notification` is persisted per recipient with:
   `recipientUserId`, `type` (assigned / mentioned / changed), a `source` reference
   (the task and, for mentions, the comment), a `read` flag, and a `createdAt` timestamp.
   It is a simple, CRUD-like **entity**, not an aggregate root with rich invariants — it
   records that something happened and whether the recipient has seen it.

2. **Generated by domain-event handlers, never inline.** Notifications are produced by
   handlers reacting to domain events, consistent with ADR-0003's event-driven,
   eventually-consistent cross-aggregate model. The emitting slices stay ignorant of
   notifications; they only publish their existing events:
   - `TaskAssigned` — emitted from the **assignment** slice (008) when an editor/owner adds
     an assignee (FR-070).
   - `UserMentioned` — emitted from the **comments** slice (009) when a comment @mentions a
     project member (FR-074).
   - `TaskChangedForAssignee` — derived from task-change events, notifying recipients who are
     assignees of the changed task (FR-079). A "task change" is not one event: it is the set
     of existing task-mutation events already published by core Task Management slices (e.g.
     002/005/010). The slice-017 handler subscribes to those existing events rather than
     relying on a single new event minted in 008/009, and resolves the assignee recipient set
     for each.
   The handlers themselves live in the **notifications** slice (017). They run under
   Wolverine and are delivered reliably through the transactional outbox (ADR-0001), so a
   notification is never lost if a handler or the broker hiccups.

3. **Fan-out is per-recipient.** A single event (e.g. a task with three assignees changing)
   produces one `Notification` row per affected recipient. The recipient set is resolved in
   the handler (assignees of the task, or the mentioned member), and the actor who caused
   the change is excluded — a user is not notified of their own action.

4. **Preference filtering before persistence.** Per-type preferences (FR-083) are checked
   in the handler: if the recipient has disabled a type, no `Notification` is created and no
   toast is sent for that recipient. Preferences are stored per user as a small enable/disable
   set keyed by type; defaults are all-enabled.

5. **Notification center (FR-080).** A CQRS read projection (ADR-0001) serves the recipient's
   notifications **newest-first** with **read/unread** state, scoped to the calling user's
   identity (per-user isolation, Principle IX / FR-065). This is a personal feed — it carries
   no project-membership authorization beyond "these are mine."

6. **Live toast over SignalR (FR-081).** When a notification is created and the recipient is
   online, it is pushed to that user's SignalR connection as a live toast, reusing the
   real-time transport decided in ADR-0007 (US-15) — the same hub, with no new infrastructure.
   Unlike ADR-0007's per-project-group fan-out, a notification toast is addressed to the
   recipient's own connection(s), not to a project group. The toast respects `prefers-reduced-motion`
   (Principle II / FR-047) and is non-blocking (Principle III). If the recipient is offline,
   the persisted `Notification` is simply waiting in the center on next visit — no transport
   retries, no queueing beyond the durable row.

7. **Mark-read / mark-all-read (FR-082).** Marking a notification read, and marking all read,
   are commands that flip the `read` flag on the caller's own notifications. Read-state
   changes are not destructive and are **not** covered by the 30-second undo (FR-040 applies
   to task/project data, not to a personal read marker).

## Consequences

- **No new infrastructure.** Notifications reuse Wolverine + the outbox (generation) and
  SignalR (toasts). The only additions are the `Notification` entity, a per-user preferences
  store, the slice-017 event handlers, and one read projection.
- **Emit-points are cross-references, not new logic.** Slice 008 (assignment) and slice 009
  (comments) already publish `TaskAssigned` and `UserMentioned` for their own reasons, and
  core Task Management slices (e.g. 002/005/010) already publish the task-mutation events that
  feed the "changed" trigger; notifications consume all of them. This keeps those slices free
  of notification concerns and keeps the dependency direction one-way (017 depends on
  008/009 and the task-mutation events, not the reverse).
- **Eventual consistency is acceptable.** As with the event-driven model (ADR-0003), the gap
  between the triggering write and the notification appearing is negligible for ~10 users; the
  outbox (ADR-0001) guarantees
  the notification eventually lands even across a restart.
- **Out-of-scope boundary holds.** No email, no push/device, no reminders (FR-084, ASM-06,
  OOS-13/14). Presence indicators and activity/audit feeds remain out of scope (OOS-15) —
  the notification center is a list of discrete events, not an activity stream.
- **Maps FR-079..FR-084 and ENT-09.** US-16's four acceptance scenarios are covered:
  generation on assign/mention/change (FR-079), newest-first read/unread center (FR-080),
  live toast when online (FR-081), mark-read / all-read (FR-082), per-type preferences
  (FR-083), all strictly in-app (FR-084).
