# ADR-0008 — Notifications

**Status:** Accepted (2026-06-14); amended (2026-06-15) for the 2026-06-15 design
review (remediation ledger B6 and the closed "changed"-trigger / self-suppression /
retention resolutions) and re-based on constitution v4.0.0.
**Builds on:** ADR-0001 (Wolverine + domain events + transactional outbox), ADR-0003 (domain
model; events as the change signal), ADR-0004/ADR-0005/ADR-0006 (Identity & Access; the
deny-by-default, dispatch-by-visibility authorization policy; sharing/membership), ADR-0007
(Real-time: the SignalR transport, which explicitly anticipates that "Notifications (US-16)
reuse the same transport but are specified in their own ADR"), ADR-0009 (API & error
contract), and constitution v4.0.0 (Architecture & Stack: "an in-app notification center plus
live SignalR toasts, generated by domain-event handlers; sources are re-authorized on
access"; Principle IX deny-by-default authorization; Principle XI privacy & retention).
**Scope:** how in-app notifications are generated, stored, delivered, dereferenced, and
managed. Realizes US-16 and FR-079..FR-084; honors FR-085/FR-086 (account-deletion purge and
retention) and FR-096/FR-101 (dereference re-authorization; ARIA-live for server-initiated
toasts); introduces the **Notification** entity (ENT-09).

## Context

The collaborative product (constitution v4.0.0) needs to close the loop on collaboration:
a member must learn when something needs their attention — when they are assigned a task,
@mentioned in a comment, or when a task they are an assignee of changes. The triggering
acts already exist as domain events in other slices (assignment in slice 008, comments and
mentions in slice 009, and task-mutation events published throughout core Task Management —
e.g. slices 002/005/010). Notifications are a **consumer** of those events, not a new source
of truth.

Three constraints shape the decision. First, **in-app only** — email and push/device
notifications and reminders are explicitly out of scope (ASM-06, OOS-13, OOS-14, FR-084);
there is no external notification transport to integrate. Second, the real-time channel
already exists: the SignalR transport decided in ADR-0007 (US-15) pushes live updates to
members viewing shared projects, so a new notification can ride the same connection as a live
toast without new infrastructure. Third, **authorization is deny-by-default and dispatched by
the source's current visibility** (Principle IX): a notification points at a task or comment
whose access can change *after* the notification was created — a member can be removed, a
project unshared, the source deleted — so the notification's source is a **live reference**,
re-checked when read, not a captured snapshot of content the recipient may no longer be
entitled to see. This is the controlling constraint added by remediation ledger B6.

## Decisions

1. **Notification entity (ENT-09), source is a live reference.** A `Notification` is
   persisted per recipient with: `recipientUserId`, `type` (assigned / mentioned / changed),
   a `source` reference (the task id and, for mentions, the comment id), a `read` flag, and a
   `createdAt` timestamp. The `source` is a **live reference** (ledger B6): the notification
   stores *which* entity it points at, never a copy of that entity's content (title, body,
   assignee names). It is a simple, CRUD-like **entity**, not an aggregate root with rich
   invariants — it records that something happened to a referenced source and whether the
   recipient has seen it.

2. **Generated by domain-event handlers, never inline.** Notifications are produced by
   handlers reacting to domain events, consistent with ADR-0003's event-driven,
   eventually-consistent cross-aggregate model. The emitting slices stay ignorant of
   notifications; they only publish their existing events:
   - `TaskAssigned` — emitted from the **assignment** slice (008) when an editor/owner adds
     an assignee (FR-070). It carries the assignment delta and the `actorUserId` (per the
     slice-008 resolution).
   - `UserMentioned` — emitted from the **comments** slice (009) when a comment @mentions a
     project member (FR-074). The @mention is a typed token referencing a User id (slice-009
     resolution); the event carries the `actorUserId` (the comment author).
   - `TaskChangedForAssignee` — derived from task-change events, notifying recipients who are
     assignees of the changed task (FR-079). A "task change" is **not** any edit and is not a
     single event: it is the **closed trigger set** of existing task-mutation events —
     **status change, due-date change, assignee change, and project-move** (the slice-017
     resolution). Low-signal edits (e.g. description typo, label tweak, priority nudge) are
     **excluded** and never generate a notification. The slice-017 handler subscribes to those
     existing events rather than relying on a single new event minted in 008/009, resolves the
     assignee recipient set for each, and **coalesces** rapid successive changes to one task so
     a burst of edits does not fan out as a storm of toasts.
   The handlers themselves live in the **notifications** slice (017). They run under
   Wolverine and are delivered reliably through the transactional outbox (ADR-0001), so a
   notification is never lost if a handler or the broker hiccups.

3. **Fan-out is per-recipient, with self-action suppression.** A single event (e.g. a task
   with three assignees changing) produces one `Notification` row per affected recipient. The
   recipient set is resolved in the handler (assignees of the task, or the mentioned member).
   The actor who caused the change is **excluded by matching `actorUserId` on the event
   envelope** against each candidate recipient — a user is never notified of their own action
   (self-assignment, self-mention, or editing a task they are assigned to). Suppression keys
   on the carried `actorUserId`, not on inference, so it is uniform across all three trigger
   types.

4. **Preference filtering before persistence.** Per-type preferences (FR-083) are checked
   in the handler: if the recipient has disabled a type, no `Notification` is created and no
   toast is sent for that recipient. Preferences are stored per user as a small enable/disable
   set keyed by type; defaults are all-enabled.

5. **Notification center (FR-080).** A CQRS read projection (ADR-0001) serves the recipient's
   notifications **newest-first** with **read/unread** state, scoped to the calling user's
   identity (per-user isolation, Principle IX / FR-065). This is a personal feed — the list of
   *one's own* notification rows carries no project-membership check, because the rows
   themselves are non-sensitive (a type, a source id, a timestamp). Access control is enforced
   not on the list but on **dereference** (Decision 6).

6. **Dereferencing a notification re-runs authorization (ledger B6 / FR-096).** Because the
   `source` is a live reference, **opening or expanding a notification re-runs the
   deny-by-default, dispatch-by-visibility authorization policy (Principle IX) against the
   source as it stands now** — current `ProjectMembership` + role for a shared-project source,
   ownership for personal data. Two consequences follow:
   - **No content beyond current authorization.** The notification payload (in the center and
     in the toast) carries no source content the recipient is not *currently* entitled to see.
     Any title/snippet shown is fetched through the same authorized read path as any other
     view, never read from a stored copy on the notification.
   - **Denied → "no longer available."** If authorization now fails — the recipient was
     removed from the project, the project was unshared, or the source was deleted — the
     notification resolves to a neutral **"no longer available"** state rather than surfacing
     leaked content or a raw error. The stale row may remain in the center as a tombstone entry
     but reveals nothing about the source. This closes the leak where a notification created
     while a user had access would otherwise expose content after access was revoked (Principle
     IX live-revocation, ledger B5/B6).

7. **Live toast over SignalR (FR-081).** When a notification is created and the recipient is
   online, it is pushed to that user's SignalR connection as a live toast, reusing the
   real-time transport decided in ADR-0007 (US-15) — the same hub, with no new infrastructure.
   Unlike ADR-0007's per-project-group fan-out, a notification toast is addressed to the
   recipient's own connection(s), not to a project group. The toast carries only the
   re-authorized, non-sensitive payload (Decision 6); it is announced to assistive technology
   via a polite ARIA-live region without stealing focus (Principle II / FR-101), respects
   `prefers-reduced-motion` (FR-047), and is non-blocking (Principle III). If the recipient is
   offline, the persisted `Notification` is simply waiting in the center on next visit — no
   transport retries, no queueing beyond the durable row.

8. **Mark-read / mark-all-read (FR-082).** Marking a notification read, and marking all read,
   are commands that flip the `read` flag on the caller's own notifications. Read-state
   changes are not destructive and are **not** covered by the 30-second undo (FR-040 applies
   to task/project data, not to a personal read marker).

9. **Retention and erasure (FR-086 / FR-085, Principle XI).** Notifications are **retained
   until the recipient's account is deleted** — there is no separate age-out, purge schedule,
   or cap (a row that resolves to "no longer available" still ages with the account, not
   sooner). This is the explicit retention stance Principle XI requires; silence is not
   acceptable. On account deletion (US-17 / FR-085), the erasure cascade **purges the deleted
   user's notification rows** outright. Notifications hold no third-party PII of their own
   (they are id references, not content), so when a *source* author or actor deletes their
   account, the referencing notifications are unaffected beyond what dereference re-auth
   already governs — a deleted or anonymized source simply resolves to "no longer available."

## Consequences

- **No new infrastructure.** Notifications reuse Wolverine + the outbox (generation) and
  SignalR (toasts). The only additions are the `Notification` entity, a per-user preferences
  store, the slice-017 event handlers, one read projection, and the dereference re-auth read
  path (which reuses the existing authorization policy, not a new mechanism).
- **Emit-points are cross-references, not new logic.** Slice 008 (assignment) and slice 009
  (comments) already publish `TaskAssigned` and `UserMentioned` — both now carrying
  `actorUserId` for self-suppression — for their own reasons, and core Task Management slices
  (e.g. 002/005/010) already publish the status/due/assignee/project-move events that feed the
  closed "changed" trigger set; the slice-017 handlers consume all of them. This keeps those
  slices free of notification concerns and keeps the dependency direction one-way (017 depends
  on 008/009 and the task-mutation events, not the reverse).
- **Authorization is enforced at dereference, not at list time.** The center can serve a
  user's own rows cheaply, and the cost (and correctness) of access control sits on opening a
  notification — exactly where source content would otherwise leak. This is consistent with
  Principle IX (deny-by-default, dispatched by current visibility) and live revocation (B5):
  losing access to a project silently turns its old notifications into "no longer available."
- **Eventual consistency is acceptable.** As with the event-driven model (ADR-0003), the gap
  between the triggering write and the notification appearing is negligible for ~10 users; the
  outbox (ADR-0001) guarantees the notification eventually lands even across a restart.
  Coalescing trades a small added delay for far fewer toasts under rapid edits.
- **Out-of-scope boundary holds.** No email, no push/device, no reminders (FR-084, ASM-06,
  OOS-13/14). Presence indicators and activity/audit feeds remain out of scope (OOS-15) —
  the notification center is a list of discrete events, not an activity stream. The "changed"
  trigger set is deliberately closed (status/due/assignee/project-move) so the center does not
  degrade into a low-signal change log.
- **Maps FR-079..FR-086, FR-096, FR-101 and ENT-09.** US-16's acceptance scenarios are
  covered: generation on assign/mention/closed-set-change with self-suppression and coalescing
  (FR-079), newest-first read/unread center (FR-080), live toast when online with ARIA-live
  (FR-081 / FR-101), mark-read / all-read (FR-082), per-type preferences (FR-083), all
  strictly in-app (FR-084); dereference re-authorization with "no longer available" (FR-096,
  ENT-09 live reference); and the retention/erasure stance (FR-086 retained-until-deletion,
  FR-085 purge on account deletion).
